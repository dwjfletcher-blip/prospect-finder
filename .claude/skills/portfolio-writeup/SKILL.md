# Skill: Portfolio Writeup

Turn a finished project into a short, client-facing case study for the portfolio site.

## When to Use

When David has completed a project (client work or personal build) and needs to document it in a way that sells his capabilities to the next prospect.

## How to Run This Skill

1. Ask for the inputs below. Ask all at once.
2. Write the case study using the output format below.
3. Always produce a short version (2-3 sentences) and a full version — the short version is for cards/previews, the full version is for the project detail page.

## Inputs Needed

- Project name
- What it is (website, web app, tool, etc.)
- Who it was built for (client type, or "personal/portfolio project" if spec work)
- The problem it solved or the goal it achieved
- What David specifically built (features, pages, functionality)
- Tech stack used
- Any measurable result or outcome (if client work — e.g., "client said bookings increased", "replaced a 2012 site")
- One thing that was technically interesting or challenging

## Output Format

---

### Short Version (for portfolio card)

**[Project Name]**
[One sentence: what it is and who it's for.]
[One sentence: the outcome or what makes it notable.]

`[Tag]` `[Tag]` `[Tag]`

---

### Full Version (for project detail page)

**[Project Name]**

**Overview**
[2-3 sentences. What the project is, who it's for, and why it exists. Written for a business owner reading a portfolio — not a developer.]

**The Problem**
[1-2 sentences. What was broken, missing, or needed before this project existed.]

**What Was Built**
- [Feature or deliverable]
- [Feature or deliverable]
- [Feature or deliverable]

**Tech Stack**
[List with one-word reason for each choice — keep it readable for non-technical clients]

**Result**
[1-2 sentences. What changed after launch. If no client data exists, describe the before/after state honestly. Don't invent metrics.]

**What Made It Interesting**
[1 sentence. One technical or design challenge worth mentioning — shows depth without jargon.]

---

## Hard Rules

- Write for business owners, not developers. Avoid jargon. If you must use a tech term, explain it in plain English in the same sentence.
- Never invent metrics. If there's no data, describe the qualitative outcome ("replaced a site that hadn't been updated since 2015").
- Every writeup needs a result — even if it's "gave the business a professional web presence for the first time."
- Keep the full version under 250 words. Prospects skim portfolios — brevity signals confidence.
- Tags should reflect what the client cares about: "Local Business", "Mobile-Responsive", "Fast Delivery" — not just tech stack names.
