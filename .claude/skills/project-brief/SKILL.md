# Skill: Project Brief

Turn a rough idea into a scoped, defined project description.

## When to Use

When David has a project idea (vague or detailed) and needs to define it clearly before building, pitching, or pricing it.

## How to Run This Skill

1. Ask David to describe the idea in his own words — as rough as it is.
2. Ask any clarifying questions needed (see below). Ask them all at once, not one at a time.
3. Generate the brief using the output format below.

## Clarifying Questions (ask only what's missing from the description)

- Who is this for? (type of business, size, industry)
- What problem does it solve for them?
- What does success look like for the client?
- Any features that are definitely in scope?
- Any features that are definitely out of scope?
- Do you have a tech stack in mind, or should I recommend one?
- Is there a deadline or target launch date?
- Is this for a specific client, or a spec project for your portfolio?

## Output Format

---

**Project Name:** [Short, descriptive name]

**One-Line Description:** [What it does and who it's for — one sentence]

**Problem It Solves:** [The specific pain point this addresses for the client]

**Target Client:** [Type of business, size, industry this is built for]

**In Scope:**
- [Feature or deliverable]
- [Feature or deliverable]
- [Feature or deliverable]

**Out of Scope:**
- [What this explicitly does not include]

**Recommended Tech Stack:** [Stack with one-line reason for each choice]

**Estimated Timeline:** [Rough range — e.g., "1-2 weeks" — with brief reasoning]

**Success Criteria:** [How you and the client know this is done and working]

**Portfolio / Pitch Notes:** [One paragraph David can use when describing this project to a prospective client or in a portfolio]

---

## Notes

- Keep scope tight. The most common mistake on early projects is underpricing because scope crept.
- If David hasn't identified a specific client, note that in the brief and flag it — a project without a target buyer is a portfolio piece, not a sale.
- Always recommend the simplest tech stack that gets the job done. David is early-stage; complexity is the enemy of shipping.
